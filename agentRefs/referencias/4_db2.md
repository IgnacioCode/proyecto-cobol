## 1. Por qué DB2 en mainframe es diferente

En motores como MySQL o PostgreSQL, muchas decisiones de almacenamiento físico son transparentes para el desarrollador: se crea una tabla y el motor decide dónde y cómo guardar los datos.

En **DB2 for z/OS** el modelo es más explícito. DB2 expone capas de almacenamiento y administración que permiten controlar:

- En qué volúmenes físicos viven los datos.
- En qué storage group se crean los datasets.
- En qué database y tablespace se ubican las tablas.
- Qué buffer pool se usa.
- Qué granularidad de locks se aplica.
- Cómo se administran índices, planes, permisos y transacciones.

Consecuencia práctica: en DB2 z/OS no alcanza con hacer `CREATE TABLE`; antes hay que definir o usar correctamente los objetos físicos/lógicos donde vivirá la tabla.

## 2. Jerarquía de objetos DB2

Jerarquía general:

```text
STORAGE GROUP
└── DATABASE
    └── TABLESPACE
        └── TABLE
            └── INDEX
```

Cada nivel tiene una responsabilidad distinta.

---

## 3. Storage Group

El **Storage Group** o `STOGROUP` define en qué volúmenes DASD puede crear datasets DB2.

En el entorno del curso, el storage group ya existe:

```sql
CREATE STOGROUP UNLAM
  VOLUMES(BIGDB5)
  VCAT DSND10;
```

Interpretación:

- `UNLAM`: nombre del storage group.
- `BIGDB5`: volumen físico donde residen datos.
- `VCAT DSND10`: catálogo asociado.

Para el TP no hay que crear ni modificar el storage group. Solo se referencia al crear tablespaces:

```sql
USING STOGROUP UNLAM
```

---

## 4. Database

En DB2 z/OS, una **database** es un contenedor lógico de tablespaces. No equivale exactamente a una base de datos completa estilo MySQL/PostgreSQL.

En el curso, la database disponible es:

```text
UNLAM
```

Al crear un tablespace:

```sql
CREATE TABLESPACE G6USU
  IN UNLAM
```

`IN UNLAM` indica que el tablespace pertenece a la database `UNLAM`.

---

## 5. Tablespace

El **tablespace** es el objeto que realmente contiene los datos de una tabla. DB2 guarda las filas en datasets físicos asociados al tablespace.

Para el TP, cada tabla debe tener su propio tablespace.

Convención:

```text
G#USU  Tablespace de USUARIOS
G#LIB  Tablespace de LIBROS
G#PRE  Tablespace de PRESTAMOS
```

Para grupo 6:

```text
G6USU
G6LIB
G6PRE
```

Ejemplo:

```sql
CREATE TABLESPACE G6USU
  IN UNLAM
  USING STOGROUP UNLAM
  PRIQTY 20
  SECQTY 10
  BUFFERPOOL BP0
  LOCKSIZE ROW
  COMPRESS NO;
```

Parámetros importantes:

| Parámetro | Significado |
|---|---|
| `IN UNLAM` | Database contenedora. |
| `USING STOGROUP UNLAM` | Storage group físico. |
| `PRIQTY` | Espacio primario inicial. |
| `SECQTY` | Espacio adicional para extensiones. |
| `BUFFERPOOL BP0` | Buffer pool de memoria. |
| `LOCKSIZE ROW` | Bloqueo a nivel fila. Mejora concurrencia. |
| `COMPRESS NO` | Sin compresión. |

## 6. Table

La tabla contiene filas y columnas, como en otros motores relacionales. La diferencia importante en DB2 z/OS es que se indica explícitamente en qué tablespace vive:

```sql
CREATE TABLE KC03G24.USUARIOS (
  COD_USUARIO    CHAR(10)     NOT NULL,
  NOMBRE         VARCHAR(30)  NOT NULL,
  APELLIDO       VARCHAR(30)  NOT NULL,
  TIPO_USUARIO   CHAR(1)      NOT NULL,
  EMAIL          VARCHAR(50)  NOT NULL,
  TELEFONO       VARCHAR(20),
  DIRECCION      VARCHAR(60),
  FECHA_ALTA     DATE,
  FECHA_BAJA     DATE,
  ESTADO         CHAR(1)      DEFAULT 'A',
  CONSTRAINT PK_USUARIOS PRIMARY KEY (COD_USUARIO),
  CONSTRAINT CK_TIPO CHECK (TIPO_USUARIO IN ('E','D','A')),
  CONSTRAINT CK_ESTADO_USU CHECK (ESTADO IN ('A','I'))
) IN UNLAM.G6USU;
```

Partes clave:

- `KC03G24.USUARIOS`: tabla `USUARIOS` bajo schema/owner `KC03G24`.
- `IN UNLAM.G6USU`: database `UNLAM`, tablespace `G6USU`.
- `PK_USUARIOS`: clave primaria.
- `CHECK`: restricciones de valores permitidos.

Nota: el schema/owner permite que distintos grupos tengan tablas con el mismo nombre sin conflictos.

## 7. Index

Los índices aceleran consultas y validan unicidad, a costo de espacio y tiempo de escritura.

Ejemplos:

```sql
CREATE UNIQUE INDEX KC03G24.IDZ_PK_USU
  ON KC03G24.USUARIOS (COD_USUARIO);
```

```sql
CREATE INDEX KC03G24.IDP_USU_TIPO
  ON KC03G24.USUARIOS (TIPO_USUARIO);
```

Índice único para email:

```sql
CREATE UNIQUE INDEX KC03G24.IDU_USU_EMAIL
  ON KC03G24.USUARIOS (EMAIL);
```

Notas:

- Cuando se dropea una tabla, sus índices se eliminan automáticamente.
- Se puede dropear y recrear un índice sin borrar la tabla.
- Para validar email único, conviene un índice único sobre `EMAIL`.

## 8. Ciclo de vida de un dato

Flujo simplificado desde un programa COBOL:

```text
Programa COBOL
  EXEC SQL INSERT INTO KC03G24.USUARIOS ...
        │
        ▼
Motor DB2 / subsistema DBDG
  1. Valida permisos
  2. Busca tabla/tablespace en catálogo
  3. Escribe página en buffer pool
  4. Registra operación en log
        │
        ▼
Buffer Pool BP0
        │
        ▼
Dataset físico en DASD
        │
        ▼
COMMIT confirma la transacción
```

Punto crítico: **COMMIT**.

Si un programa COBOL DB2 hace `INSERT`, `UPDATE` o `DELETE`, los cambios no quedan confirmados definitivamente hasta ejecutar:

```cobol
EXEC SQL COMMIT END-EXEC.
```

Si se ejecuta:

```cobol
EXEC SQL ROLLBACK END-EXEC.
```

DB2 revierte la transacción.

## 9. STEPLIB en jobs DB2

Para ejecutar SQL o programas COBOL DB2, los jobs deben incluir librerías DB2.

Ejemplo:

```jcl
//STEPLIB  DD DSN=DSND10.SDSNLOAD,DISP=SHR
//         DD DSN=DSND10.DBDG.SDSNEXIT,DISP=SHR
//         DD DSN=DSND10.DBDG.RUNLIB.LOAD,DISP=SHR
```

Rol de cada dataset:

| Dataset | Rol |
|---|---|
| `DSND10.SDSNLOAD` | Librería principal de DB2. |
| `DSND10.DBDG.SDSNEXIT` | Exits específicos del subsistema `DBDG`. |
| `DSND10.DBDG.RUNLIB.LOAD` | Runtime del subsistema; contiene utilidades como `DSNTEP2`. |

Si falta una librería, pueden aparecer errores como programa no encontrado o fallas al iniciar DB2.

## 10. DSNTEP2

`DSNTEP2` es una utilidad IBM para ejecutar SQL desde JCL.

Lee sentencias SQL desde `SYSIN`, las ejecuta contra DB2 y muestra resultados/mensajes en spool.

Uso típico:

- `CREATE TABLESPACE`.
- `CREATE TABLE`.
- `CREATE INDEX`.
- `DROP`.
- `GRANT`.
- Consultas de diagnóstico.

Ejemplo:

```jcl
//STEP01   EXEC PGM=IKJEFT01,DYNAMNBR=20
//STEPLIB  DD DSN=DSND10.SDSNLOAD,DISP=SHR
//         DD DSN=DSND10.DBDG.SDSNEXIT,DISP=SHR
//         DD DSN=DSND10.DBDG.RUNLIB.LOAD,DISP=SHR
//SYSTSPRT DD SYSOUT=*
//SYSPRINT DD SYSOUT=*
//SYSTSIN  DD *
  DSN SYSTEM(DBDG)
  RUN PROGRAM(DSNTEP2) PLAN(DSNTEP13)
  END
/*
//SYSIN    DD *
  SELECT CURRENT DATE
    FROM SYSIBM.SYSDUMMY1;
/*
```

Explicación:

| Sentencia | Significado |
|---|---|
| `DSN SYSTEM(DBDG)` | Conecta al subsistema DB2 `DBDG`. |
| `RUN PROGRAM(DSNTEP2)` | Ejecuta utilidad SQL. |
| `PLAN(DSNTEP13)` | Plan autorizado para DSNTEP2. |

## 11. Plan DB2 y BIND

Un **plan DB2** contiene información de acceso optimizada para ejecutar SQL.

En COBOL con SQL embebido:

1. El programa se precompila/compila con SQL.
2. Se genera un DBRM.
3. Se ejecuta `BIND`.
4. DB2 crea o reemplaza el plan.
5. El programa se ejecuta usando ese plan.

Ejemplo:

```jcl
//SYSTSIN  DD *
  DSN SYSTEM(DBDG)
  BIND PLAN(GRUPO6) MEMBER(USUMANT) -
       ACTION(REPLACE) ISOLATION(CS) ENCODING(EBCDIC)
  RUN PROGRAM(USUMANT) PLAN(GRUPO6) -
      LIB('KC02814.GRUPO6.LOAD.LIBRARY')
  END
/*
```

Notas:

- Sin `BIND`, el programa no puede ejecutar SQL.
- Si cambia el SQL embebido, hay que recompilar y bindear.
- El plan usado en `RUN PROGRAM(...) PLAN(...)` debe existir y estar autorizado.

## 12. Permisos y GRANT

Los usuarios necesitan permisos sobre tablas, secuencias y planes.

Ejemplo sobre tablas:

```sql
GRANT SELECT, INSERT, UPDATE, DELETE
  ON TABLE KC03G24.USUARIOS
  TO KC03G28, KC03G2B;
```

Ejemplo sobre secuencia:

```sql
GRANT USAGE
  ON SEQUENCE KC03G24.SEQ_PRESTAMOS
  TO KC03G28, KC03G2B;
```

Ejemplo sobre plan:

```sql
GRANT EXECUTE
  ON PLAN CURSOG06
  TO KC03G28, KC03G2B;
```

`GRANT EXECUTE ON PLAN` permite que otros usuarios ejecuten programas que usan ese plan.

## 13. Convenciones de nombres del TP

Para evitar conflictos, cada grupo usa su número y un owner específico.

Convenciones:

| Objeto | Convención | Ejemplo grupo 6 |
|---|---|---|
| Tablespace usuarios | `G#USU` | `G6USU` |
| Tablespace libros | `G#LIB` | `G6LIB` |
| Tablespace préstamos | `G#PRE` | `G6PRE` |
| Schema/owner | `KC03XXX` | `KC03G24` |
| Tabla usuarios | `KC03XXX.USUARIOS` | `KC03G24.USUARIOS` |
| Tabla libros | `KC03XXX.LIBROS` | `KC03G24.LIBROS` |
| Tabla préstamos | `KC03XXX.PRESTAMOS` | `KC03G24.PRESTAMOS` |
| Secuencia | `KC03XXX.SEQ_PRESTAMOS` | `KC03G24.SEQ_PRESTAMOS` |
| Índices | `KC03XXX.ID...` | `KC03G24.IDP_USU_TIPO` |

El owner es el usuario que crea los objetos. Ese userid queda como qualifier/schema.

## 14. SQLCODEs comunes

| SQLCODE | Significado | Causa probable |
|---:|---|---|
| `0` | Éxito | Operación correcta. |
| `+100` | No se encontraron filas | `WHERE` sin resultados. |
| `-204` | Objeto no definido | Tabla/secuencia inexistente o mal qualifier. |
| `-206` | Columna inexistente | SQL no coincide con DDL real. |
| `-407` | NULL en columna `NOT NULL` | Falta campo obligatorio. |
| `-530` | Violación de foreign key | Valor referenciado no existe en tabla padre. |
| `-545` | Violación de check constraint | Valor no cumple `CHECK`. |
| `-803` | Clave duplicada | Ya existe PK o índice único. |
| `-811` | Más de una fila en `SELECT INTO` | Consulta devuelve múltiples filas. |
| `-904` | Recurso no disponible | Tablespace/lock/recurso no disponible. |
| `-911` | Deadlock o timeout | Transacciones bloqueadas entre sí. |
| `-922` | No autorizado | Falta permiso sobre tabla, plan u objeto. |

Nota práctica: en COBOL/mainframe los números con signo pueden imprimirse con signo sobreimpreso. Por ejemplo, un `SQLCODE -204` puede verse con un carácter final raro si se muestra sin editar correctamente.

## 15. Secuencia para crear objetos del TP

Orden correcto:

```text
1. Tablespaces
   G6USU, G6LIB, G6PRE
        ↓
2. Tablas
   USUARIOS, LIBROS, PRESTAMOS
        ↓
3. Secuencia
   SEQ_PRESTAMOS
        ↓
4. Índices
   PK, búsqueda, unicidad
        ↓
5. GRANTs
   Permisos al resto del grupo
```

Orden de eliminación:

```text
1. Índices si se dropean manualmente
2. Tablas, respetando foreign keys
3. Secuencia
4. Tablespaces, si corresponde
```

En DB2, al dropear una tabla, sus índices asociados se eliminan automáticamente.

Si falla la creación del entorno, la guía recomienda usar un JCL tipo `DROPALL` antes de volver a ejecutar `RUNSQL`.

## 17. Consultas de diagnóstico

Ver columnas reales de una tabla:

```sql
SELECT NAME, COLTYPE, LENGTH, NULLS
FROM SYSIBM.SYSCOLUMNS
WHERE TBCREATOR = 'KC03G24'
  AND TBNAME = 'USUARIOS'
ORDER BY COLNO;
```

Ver tablas del schema:

```sql
SELECT CREATOR, NAME, TYPE
FROM SYSIBM.SYSTABLES
WHERE CREATOR = 'KC03G24'
ORDER BY NAME;
```

Ver índices:

```sql
SELECT CREATOR, NAME, TBNAME, UNIQUERULE
FROM SYSIBM.SYSINDEXES
WHERE TBCREATOR = 'KC03G24'
ORDER BY TBNAME, NAME;
```

Ver permisos sobre tabla:

```sql
SELECT GRANTEE, SELECTAUTH, INSERTAUTH, UPDATEAUTH, DELETEAUTH
FROM SYSIBM.SYSTABAUTH
WHERE TCREATOR = 'KC03G24'
  AND TTNAME = 'USUARIOS';
```

## 18. Buenas prácticas

- Calificar siempre tablas con owner/schema: `KC03G24.USUARIOS`.
- Alinear COBOL SQL embebido con el DDL real.
- Confirmar columnas con `SYSIBM.SYSCOLUMNS` ante `SQLCODE -206`.
- Hacer `COMMIT` explícito en programas COBOL DB2.
- Usar `ROLLBACK` al detectar error SQL.
- Crear tablas antes que índices.
- Crear tablas padre antes que tablas hijas con foreign keys.
- Dropear en orden inverso cuando existan dependencias.
- Recompilar y bindear si cambia SQL embebido.
- Otorgar `GRANT EXECUTE ON PLAN` si otros usuarios ejecutan el programa.
- Otorgar permisos sobre tablas y secuencias al resto del grupo.
